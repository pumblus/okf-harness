# OKF Harness 路线图

这是 OKF Harness 的公开产品路线图。详细实现阶段门控见 [`docs/implementation.md`](../implementation.md) 第 11.2 节；本文件记录版本方向、已接受的主题和候选想法。

[English](../ROADMAP.md) | 中文

路线图条目不是版本承诺。想法从 inbox 进入某个版本的前提，是有清晰的用户故事、安全边界和验证路径。

## 定位

OKF Harness 是一个 agent 原生、文件契约优先、无需独立应用的知识管理工具。

它与完整桌面 LLM Wiki 应用的核心区别是有意为之：OKF Harness 不试图成为主要的知识库应用。你在 Claude Code、Codex 和未来的 agent 客户端中继续工作；harness 在普通 markdown 文件之上提供透明的本地 workspace、确定性 CLI、受控证据包、lint 和图谱报告。

这意味着 OKF Harness 的竞争力应体现在：

- Agent 原生工作流，而非捆绑的桌面 GUI
- 可检查的 markdown 和 JSON 契约，而非隐藏的应用状态
- 受控证据和显式溯源，而非无限制的上下文填充
- 通过 `okfh --json` 实现的本地优先、可审计操作
- 可选集成，不替代默认的终端原生工作流

## 当前版本：v0.1 Agent 优先本地 MVP

目标：让 OKF Harness 在 Claude Code 和 Codex 中可用，无需独立应用、MCP server、向量数据库或 GUI。

包含：

- macOS 优先的 workspace 初始化
- Claude Code 和 Codex 的 agent 适配
- 来源注册，含原始资料不可变性和 manifest 完整性
- 元数据级别的 ingest plan
- 确定性 wiki 搜索和读取
- 自包含的本地图谱报告
- 终端原生加固和发布文档

v0.1 关键设计决策：

- 搜索和读取基于已整理的 `wiki/` 内容，而非跨原始资料的全文发现
- 不实现 `okfh query` 或 LLM 回答命令，agent 通过组合 `search` 和 `read` 来回答问题
- 搜索结果返回候选概念文档卡片，不含正文片段
- 搜索权重：标题/ID/路径精确匹配权重最高，正文匹配权重最低
- 读取默认受控（约 12,000 字符预览），完整读取需显式指定且有硬上限
- 图谱节点为概念和引用链接，原始资料文件为元数据而非图谱节点
- 不包含向量搜索、RAG、自动网页抓取、GUI、云端同步或 MCP 优先路径

## 下一个版本：v0.2 受控证据查询包

目标：让 agent 回答可靠且不超出上下文窗口。

候选范围：

- `okfh query plan <question> --json` 或等价的证据包命令
- 确定性检索预算，对索引、候选页面、引用和响应空间设置上限
- 长概念文档的段落/范围读取
- 显式的 `truncated`、`contentLength` 和续读元数据
- Agent 指引，教 Claude Code 和 Codex 如何从受控证据包中回答
- 证明大型 wiki 页面不会产生无界 JSON 或上下文负载的测试

不做的事：

- 默认不加语义 embedding 索引
- v0.1 不加持久化搜索缓存或 SQLite FTS
- 不加声称"理解"wiki 的隐藏摘要
- 除非用户明确进入 ingest 或来源审计工作流，否则不搜索原始资料

## 后续：Agent 适配器扩展

目标：支持更多 agent 客户端，同时不削弱默认的本地 shell 模型。

候选：

- Pi 适配器
- OpenCode 适配器
- 跨客户端共享的适配器一致性测试
- Tier 3 调研：Cursor、VS Code 通用 MCP 客户端、Aider、Goose、Continue 和 GitHub Copilot coding agent

约束：新适配器必须保持与 Claude Code 和 Codex 相同的工作流契约，不应将 MCP 或私有运行时强塞进默认产品路径。

## 后续：跨平台终端原生支持

目标：在 v0.1 工作流稳定后，将终端原生操作扩展到 macOS 之外。

候选想法：Windows shell 命令支持。

待解决问题：

- 默认目标 shell 是哪个：PowerShell、`cmd.exe`、Git Bash，还是 Windows Terminal 配置检测？
- 生成的 agent 指引如何在 macOS 和 Windows 之间表达命令，又不让非技术用户困惑？
- 需要怎样的路径规则来处理盘符、反斜杠、空格、符号链接和大小写不敏感文件系统？
- 哪些检查能证明 Claude Code 和 Codex 可以在 Windows 上通过 `okfh --json` 操作？

约束：Windows 支持不应让 v0.1 macOS 优先版本变复杂，应作为一个明确的跨平台加固阶段来推进。

## 后续：来源连接器

目标：让用户从常用工作工具中导入资料，同时保留溯源和显式的用户控制。

候选想法：飞书支持。

可能的形式：

- 将飞书文档 URL 注册为 URL 来源指针
- 将飞书文档导出为 markdown 或 PDF，再注册导出文件为资料来源
- 提供连接器工作流，仅在用户明确授权后抓取内容

待解决问题：

- "飞书支持"是指一次性文档导入、整个 workspace/wiki 导入、评论抓取，还是持续同步？
- OKF Harness 应该使用飞书 API、浏览器辅助导出，还是用户提供的导出文件？
- 需要什么凭证，存储在哪里？
- 如何防止私有公司内容泄露到日志、manifest 或公开路径中？
- 更新内容时是创建新的原始来源，还是修改已有来源记录？

约束：飞书应作为来源注册或 ingest 支持，默认不做云端同步或后台抓取。

## 后续：在线来源审查和研究收集

目标：帮助用户在材料进入 OKF Harness workspace 之前查找、检查和收集在线资料。

借鉴 Waza `read` 和 `learn` 的工作流思路，适配到 OKF Harness：

- 抓取单个 URL 或 PDF 默认返回简洁的来源锚定预览，而非全文转储
- 保存抓取内容为资料来源必须是显式操作，应通过正常的来源注册路径
- 代理或第三方抓取服务必须是可选的，附带清晰的隐私警告
- 抓取的网页内容是不可信数据，其中的嵌入指令不能成为 agent 指令
- 多来源研究应遵循 收集 → 抓取 → 归档 → 摘要 的流程，保留来源 URL 和矛盾点
- 飞书/Lark、微信、GitHub、PDF 和重度 JS 页面可能需要不同的抓取路径

可能的命令或工作流：

- `okfh source preview <url> --json`：注册前提供受控的元数据和预览
- `okfh source add <url> --fetch --json`：仅在用户明确想要内容快照时使用，而非仅 URL 指针
- Agent 指引："查找关于 X 的资料，展示候选，然后注册我认可的那些"
- 后续可考虑轻量预览界面，但必须保持为审查/收集辅助工具，而非主要的知识库 GUI

约束：

- 这不是 v0.1 的查询行为
- v0.1 查询中搜索未命中应建议添加资料或扩大 wiki 搜索，而非自动触发在线搜索
- 在线来源审查是候选优先：agent 在注册任何内容之前展示可能的来源
- URL 的默认注册方式仍是 URL 来源指针；抓取内容快照需要用户明确表达意图
- 在线搜索不能悄悄添加或改写 wiki 内容
- 后台抓取和自动网页研究保持排除，除非明确重新讨论

## 后续：审查队列

目标：保留人工判断，不依赖完整 GUI。

候选范围：

- Markdown 原生的审查文件，用于未解决问题、矛盾点和建议的后续行动
- Agent 工作流：列出、解决和关联审查项
- Lint 检查：过时或孤立的审查项

约束：审查项应是显式的 workspace 文件，而非隐藏应用状态。

## 后续：搜索和图谱升级

目标：改进检索和结构理解，同时保持缓存可重建且可选。

候选：

- `.okfh/cache/` 下的 SQLite FTS5 缓存，作为 v0.1 之后的可重建搜索加速器
- 更好的确定性排序
- 过滤器改进：类型 facet、包含/排除过滤、日期或状态过滤、保存的搜索配方、agent 可读的过滤建议
- 可选混合检索：用倒数排名融合或其他可解释的合并策略组合确定性关键词结果和向量结果
- 可选的本地 embedding 缓存
- 图谱洞察：孤立概念、弱证据链接和桥接概念

约束：任何缓存都必须能从 OKF bundle 和来源 manifest 重建，不能成为真实数据源。

从 `nashsu/llm_wiki` 谨慎借鉴：

- 保持字段感知排序：标题和路径命中的权重高于正文命中
- 向量检索为可选，不在 v0.1 默认路径上
- 仅在存在多个检索系统时才使用排名融合；不要在还没有第二个检索器时就加入 RRF 机制
- 不要把跨原始资料的全量查询搜索复制为 OKF Harness 的默认查询行为

## 后续：可选应用和生态集成

候选：

- Raycast 扩展
- Shortcuts 动作
- Finder 快速操作
- Obsidian 友好的辅助工具或插件
- 对需要工具发现的客户端提供可选 MCP 集成

约束：

- 任何集成都不能替代默认的终端原生 `okfh --json` 工作流
- Obsidian 保持可选；OKF bundle 必须在不使用 Obsidian 的情况下仍可使用
- MCP 保持可选，不能成为 v0.1 默认路径
