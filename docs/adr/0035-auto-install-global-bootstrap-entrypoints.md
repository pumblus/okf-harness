# Auto-install global bootstrap entrypoints

OKF Harness should give people a discoverable prompt-first setup path before a workspace exists. The CLI package may therefore install managed global bootstrap entrypoints named `okf-harness-bootstrap` for supported agent clients during normal setup, while the high-frequency workspace-local entrypoint remains `okf-harness` for check, ingest, answer, and graph work after a workspace is selected. This favors explicit scope over one name everywhere: bootstrap is rare enough that the longer name is acceptable, while the common workspace workflow keeps the shorter name. The global bootstrap entrypoint should stay thin: setup, workspace discovery or selection, current-agent workspace guidance repair, and handoff into a workspace-local session, not full check, ingest, answer, or graph workflows. During first workspace setup, bootstrap should infer the workspace display name, target folder, current agent, and Git choice from the person's natural-language request, then ask only for missing or ambiguous inputs before running the initialization command; Git is a required confirmation when not explicitly stated, with no as both the default and recommended answer, and `--git` is used only when the person chooses Git. During package installation, OKF Harness should first detect installed supported agent clients, then bootstrap each detected client independently; detected clients that cannot be bootstrapped should produce a natural-language next step, while missing clients should not be treated as failures. A small `okfh bootstrap` CLI surface should exist for install, status, repair, and uninstall so package install, doctor, and agents share one deterministic tool path without making bootstrap commands the README primary workflow. Once a specific workspace is selected, bootstrap should repair workspace-local guidance only for the current agent client unless the person explicitly asks for additional clients. Bootstrap target paths must come from official agent-client documentation or verified client discovery behavior, with OS-specific home directories resolved to concrete local paths rather than guessed string templates. If bootstrap installation cannot complete during package installation, the package install should still succeed and report a next step that `okfh doctor --json` can verify or repair.
