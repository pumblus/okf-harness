# 源演化生命周期原型（一次性）

为 [#60](https://github.com/pumblus/okf-harness/issues/60) 而建的一次性原型，属于地图
[Map the invisible knowledge-maintenance loop](https://github.com/pumblus/okf-harness/issues/54)。
**不是生产代码。** 无测试、无错误处理、无持久化——问题一旦回答完就删掉。

## 要回答的问题

一个小型的、公开的双源场景，能否让所提议的生命周期、自动路径、停顿行为、源血缘和最终答案面，
透过单一的 prompt-first 入口变得可理解、可使用？

## 建模了什么

`lifecycle.mjs` 是纯逻辑那一半——一个 reducer 加若干 selector，无 I/O——编码了这些决策：
[#55](https://github.com/pumblus/okf-harness/issues/55)（六个阶段）、
[#57](https://github.com/pumblus/okf-harness/issues/57)（停顿契约、两跳封印）、
[#58](https://github.com/pumblus/okf-harness/issues/58)（零提案状态、检查点骑在阶段 5 上）、
[#59](https://github.com/pumblus/okf-harness/issues/59)（三条轴）、
[#66](https://github.com/pumblus/okf-harness/issues/66)（恢复动词）。

界面刻意分成**已存储**和**实时计算**两栏，因为这个切分本身就是被测的东西：已存储的只有无法从基底
重算的部分（manifest 行、血缘、悬空标记位、检查点）；血缘怀疑、封印、时效性都是纯函数，每帧重算。

## 场景

两版虚构的 `远程办公制度.md`，每一类 finding 各一条：

| 发现 | 有信号？ | 可读？ | 预期 |
|---|---|---|---|
| 团队负责人须公布驻场日 | 是 | 是 | 自动（补充） |
| 上限 2 → 3 天 | 是 | 取决于 `[s]` | 自动取代，或停顿 |
| 对周五闭馆只字未提 | 是 | 否 | 停顿（疑似删除） |
| 客服团队豁免 | **否** | 否 | **停不下来**——只剩报告 |

最后一行是最锋利的：它只存在于用户脑子里，所以任何停顿都不可能发生，完成报告是唯一的防线。

## 运行

```bash
pnpm proto:evolution
```

必须在真实终端里跑（raw mode 逐键读取），不能通过管道或非交互环境。

## 建议路线

1. `[1]` 交出 v1，然后连按 `空格` 走完六个阶段。全程零停顿。
2. `[a]` 提问——v1 的主张被送达。
3. `[2]` 交出 v2，`空格` 到阶段 2。同名不同哈希 → 怀疑触发 → 身份停顿。
   先按 `[i]`：问题会收窄而不是被解决。然后 `[y]`。
4. `[空格]` 到阶段 3 → 周五那个停顿。事先按 `[s]` 让上限变更变可读，看那个停顿消失。
5. `[空格]` 走完 4–6，然后 `[h]`：报告里会点名那条谁也停不住的客服豁免。
6. `[x]` 腐蚀 v1 的字节，然后 `[a]` 和 `[o]`：概念被封，第三跳的概念不被封。
7. `[m]` 砸坏 manifest：封印扩到全工作区，因为算链条用的那张地图没了。
